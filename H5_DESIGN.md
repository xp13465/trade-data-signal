# H5 移动端样式设计方案（待确认）

> 目标：移动端访问自动切换到 H5 布局，PC 端完全不受影响。
> 本文件基于现有 PC 端代码（`web/index.html` + `style.css` + `app.js`）调研后给出 3 套布局方案，**先确认方案再实施**。

---

## 一、移动端识别方式（自动切换）

| 方式 | 说明 | 采用 |
|---|---|---|
| **CSS `@media`** | `@media (max-width: 768px)` 视口驱动，手机屏自动套用 H5 样式，PC 屏（>768）自动用 PC 样式。零 JS、零用户操作、PC 零影响。 | ✅ 主用 |
| JS `body.h5` | JS 检测屏宽 ≤768 给 `<body>` 加 `.h5` class，配合需 JS 结构调整的方案（如底部导航）。 | ⚠️ 仅方案 B 需要 |
| UA 嗅探 | `navigator.userAgent` 判断手机。平板/折叠屏不可靠，不采用。 | ❌ |

**结论**：三个方案都以 `@media (max-width: 768px)` 为切换机制 → 天然"自动切换 + PC 不影响"。区别只在**布局重构程度**。

---

## 二、现状痛点（PC 端在手机上的 8 个问题）

1. **tab 栏挤不下**：`nav.tabs` 一行放 4 个主 tab + 右端 `.periods` 5 个周期按钮 = 9 个按钮，375px 屏严重挤压换行。
2. **边距过宽**：`header`/`.tabs`/`main` 的 `padding: 14-24px`，手机两侧留白占比过大。
3. **多列网格挤压**：`.industry-grid` 3 列、`.sig-items` 4 列、`.ov-2col` 2 列、`.spark-grid` `minmax(200px)`，手机单格过窄文字截断。
4. **表格溢出**：`.futures-summary-table`/`.accuracy-table`/`.ma-table` 固定列宽，手机横向溢出导致整页可横向滑动。
5. **多层 sticky 占垂直空间**：tab 栏(sticky) + `.rule-bar`(sticky) + `.industry-anchor-bar`(sticky) 叠加，手机竖屏顶部被占满。
6. **chart-hint 回测数据乱**：`.hint-row` 多列 `flex-wrap`，手机换行后买卖点/胜率/凯利数据错位难读。
7. **浮动按钮 + 弹窗未适配**：`.back-to-top`/`.rule-float-btn` 右下角，`.modal`/`.rule-modal` 宽 360-820px，手机溢出或挡内容。
8. **ECharts 边距未缩**：图表 `grid.left/right` 按 PC 设定，窄屏坐标轴标签被切。

## 三、方案 A：响应式轻适配（最小改动）

**思路**：纯 CSS `@media` 覆盖，JS 几乎不动。保留 PC 的"顶部 tab 栏"结构，只做窄屏适配（横滚 + 单列 + 表格横滚）。

```
┌──────────────────────────┐
│ 📊 情绪数据复盘看板        │  header padding 24→12
├──────────────────────────┤
│ 概览 大盘 情绪 行业  →     │  tabs overflow-x:auto 横滚
├──────────────────────────┤
│ 1月 3月 6月 1年 全部       │  .periods 移出 tab 栏，独立 sticky 行
├──────────────────────────┤
│ [一句话总结横幅]           │  全宽
│ ┌KPI┐ ┌KPI┐               │  .cards 2 列
│ └───┘ └───┘               │
│ ┌spark┐ ┌spark┐           │  .spark-grid 2 列(minmax 150px)
│ └────┘ └────┘             │
│ ┌──图表卡(全宽)──┐         │  .ov-2col 单列
│ └────────────────┘        │
│ ┌位置感┐ ┌新高低┐          │
│ └─────┘ └─────┘           │
│ ┌表格 → 横向滚动┐          │  overflow-x:auto
│ └──────────────┘          │
│                  [↑][📖]  │  浮动按钮
└──────────────────────────┘
```

**布局要点**
- `header`/`.tabs`/`main` padding → `12px`
- `.tabs` 加 `overflow-x:auto; white-space:nowrap`，隐藏滚动条；`.periods` 用 `order` 或 JS 移到 tab 栏下方独立行（sticky）
- 所有网格单列或 2 列：`.industry-grid` 2 列、`.sig-items` 2 列、`.spark-grid` `minmax(150px,1fr)`、`.cards` 2 列
- 表格父容器 `overflow-x:auto`，表格保留原结构横向滚
- `.chart-hint` `.hint-row` 改纵向堆叠（`flex-direction:column`）
- `.modal`/`.rule-modal-body` → `width:94vw`，`.summary-history-body` 全宽
- 浮动按钮位置不变（右下角手机也 OK）

**改动量**：仅 `style.css`（双前端各一份，内容一致）加一段 `@media`，JS 不改或仅 1-2 行（周期条位置）。
**优点**：改动最小、风险最低、PC 零影响、维护一套代码。
**缺点**：仍是"PC 缩小版"，非原生 App 体验；顶部 tab 横滚需手指操作，不如底部导航顺手。

## 四、方案 B：App 风格底部导航（重构，最贴合手机）

**思路**：JS 检测屏宽加 `body.h5`，主 4 tab 移到底部固定导航栏（App 风格），顶部精简为"当前页 + 时效"，周期选择器移到内容区顶部段控。

```
┌──────────────────────────┐
│ 概览        7月9日收盘分析 │  精简 header：当前 tab 名 + 时效
│ 📜更多                     │
├──────────────────────────┤
│ [一句话总结横幅]           │
│ ┌────────────────────┐    │
│ │1月 3月 6月 1年 全部 │    │  周期段控 sticky(顶下)
│ └────────────────────┘    │
├──────────────────────────┤
│ ┌KPI┐                     │  全单列流式
│ │   │                     │
│ ┌spark┐                   │
│ ┌──图表卡(全宽)──┐         │
│ ┌位置感┐ ┌新高低┐          │  窄屏仍可 2 列小卡
│ ┌表格→横滚┐                │
│ ...                       │
│                  [↑][📖]  │  浮动按钮上移避底导航
├──────────────────────────┤
│ 📊概览 │ 📈大盘 │ 😊情绪 │ 🏭行业 │  底部固定导航(4 tab)
└──────────────────────────┘
```

**布局要点**
- 顶部 `nav.tabs` 隐藏（PC 的 tab 栏），新建底部固定 `.h5-bottomnav`（4 个 tab 按钮 + 图标），复用原 `data-tab` 切换逻辑（不重写交互）
- 顶部 header 改为：左当前 tab 名、右一句话总结时效 + 📜更多入口
- `.periods` 周期段控移到内容区顶部 sticky 行（top = 精简 header 高度）
- `main` 底部 `padding-bottom:64px` 避开底部导航
- 内容单列为主，KPI/spark 等 2 列
- 浮动按钮 `bottom` 上移到 `88px`（避开 56px 底导航）
- 表格横滚、弹窗近全屏、chart-hint 堆叠（同方案 A）

**改动量**：`style.css` + `app.js`（双前端各两份）。JS 加：屏宽检测 `body.h5`、底部导航 DOM 注入或 CSS 显隐、周期条重定位、ECharts resize 复核。
**优点**：最符合手机操作习惯（拇指触达底部 tab），原生 App 体验，顶部空间释放给内容。
**缺点**：改动最大，双前端 JS 都要同步改，底部导航与现有 sticky 逻辑需协调；测试成本高。

## 五、方案 C：顶部横滚 + 周期折叠（折中）

**思路**：`@media` 切换，tab 栏保留顶部但横滚，周期选择器折叠为右上角下拉，表格转卡片视图。介于 A/B 之间。

```
┌──────────────────────────┐
│ 📊情绪看板         周期▾  │  header + 周期下拉
├──────────────────────────┤
│ 概览 大盘 情绪 行业  →     │  tabs 横滚
├──────────────────────────┤
│ [一句话总结]               │
│ ┌KPI┐ ┌KPI┐               │
│ ┌spark┐ ┌spark┐           │
│ ┌图表卡(全宽)┐             │
│ ┌期货净多空(卡片化)┐       │  表格→卡片堆叠
│   IF  净多 +1200           │
│   IC  净空 -800            │
│ ...                       │
│                  [↑][📖]  │
└──────────────────────────┘
```

**布局要点**
- tab 栏横滚（同 A）
- `.periods` 折叠为 header 右侧下拉 `<select>` 或弹层，省一行垂直空间
- 表格转卡片：每行一个卡片（`flex` 横排 label-value），避免横向滚动
- 其余同 A

**改动量**：`style.css` + 少量 `app.js`（周期下拉控件、表格卡片化渲染分支）。
**优点**：比 A 更省顶部空间，比 B 改动小；表格卡片化手机阅读更顺。
**缺点**：表格卡片化需为每个表格写卡片渲染分支，双前端重复；周期下拉比段控多一次点击。

---

## 六、三方案对比

| 维度 | A 响应式轻适配 | B App底部导航 | C 横滚+周期折叠 |
|---|---|---|---|
| 手机体验 | 中（PC缩小版） | 优（原生App） | 良 |
| 改动量 | 小（纯CSS） | 大（CSS+JS重构） | 中（CSS+少量JS） |
| PC 影响 | 零 | 零（@media隔离） | 零 |
| 双前端同步成本 | 低（CSS复制） | 高（JS逻辑同步） | 中 |
| 风险 | 低 | 中高 | 中 |
| 维护 | 一套代码 | 底部导航需持续维护 | 表格卡片分支维护 |

---

## 七、推荐

**推荐方案 B**，理由：用户明确"大多数用户用手机更方便"，底部导航是手机最自然的交互；一次重构到位，避免后续再改。若想低风险先行，可**先做 A 验证效果，再升级到 B**（A 是 B 的子集，不浪费）。
